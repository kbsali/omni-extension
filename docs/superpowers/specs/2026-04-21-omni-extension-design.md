# Omni Extension — Design Spec

**Date:** 2026-04-21
**Status:** Approved design
**Author:** Kevin Saliou (via brainstorming session)

## 1. Purpose

A Chrome extension (Manifest V3) that hosts multiple self-contained feature modules under one popup UI — inspired by SuperLens-style multi-tool extensions. The extension is designed so additional modules can be added by dropping in a single folder and registering it, without touching shared code.

The first shipped module is **Dark Mode**: a per-site dark-mode toggle that persists across browser sessions, with a global default fallback and a global brightness control.

## 2. Scope (v1)

**In scope:**

- MV3 extension scaffolding with Svelte 5 + Vite + CRXJS
- Module registry pattern supporting future feature tabs
- Dark Mode module:
  - Per-site mode (`dark` / `light` / `default`) stored by eTLD+1
  - Global `defaultMode` fallback
  - Global brightness slider (50%–100%)
  - Automatic application on matched sites at `document_start` via dynamic content script registration
  - Popup UI matching the reference screenshot (moon icon, toggle, two buttons, slider)
- `chrome.storage.sync` persistence with schema versioning
- Vitest unit tests with mocked `chrome.*` APIs

**Out of scope for v1:**

- Other modules (Cookies, Smiley keyboard, GDPR, Youtube customizer) — scaffolding supports them; implementations come later as separate specs
- Dark-Reader-style dynamic color analysis (deferred; v1 uses simple CSS filter invert)
- Playwright E2E tests (deferred per user preference for personal projects)
- Keyboard shortcuts, action badge indicators
- Subdomain-specific per-site rules (eTLD+1 granularity only)

## 3. Architecture

### 3.1 Layers

```
Core      → shared primitives (storage, domain matching, module registry)
Modules   → self-contained features (each owns UI, content script, service, storage slice)
Background → service worker orchestrator (iterates modules, wires hooks)
Popup     → Svelte app that iterates the registry to render tabs
```

### 3.3 Module Contract

Every module exports this shape from `src/modules/<name>/index.ts`:

```ts
import type { ComponentType } from 'svelte';

export interface OmniModule {
  id: string; // unique, e.g. 'dark'
  label: string; // tab label, e.g. 'Dark'
  icon: string; // emoji or inline SVG string
  Popup: ComponentType; // Svelte component shown in its tab
  onBackground?: (ctx: BackgroundCtx) => void;
  storageDefaults: Record<string, unknown>; // merged into initial storage for modules[id]
}

export interface BackgroundCtx {
  storage: TypedStorage; // from core/storage
  onStorageChange: (key: string, cb: (value: unknown) => void) => void;
}
```

Registry (`src/core/registry.ts`):

```ts
import dark from '../modules/dark';

export const modules: OmniModule[] = [dark];
```

Adding a new module: create `src/modules/<name>/`, implement `OmniModule`, import + append in `registry.ts`. No other changes required.

## 4. Storage Schema

Single root key `omni` in `chrome.storage.sync`, namespaced per module:

```ts
type Mode = 'dark' | 'light' | 'default';

interface OmniStorage {
  version: 1; // for future migrations
  modules: {
    dark: DarkStorage;
    // future module slices go here
  };
}

interface DarkStorage {
  defaultMode: 'dark' | 'light'; // global fallback
  brightness: number; // 0.5–1.0 (shown as 50%–100%)
  sites: Record<string, Mode>; // eTLD+1 → explicit override
}
```

**Defaults on first install:**

```ts
{ version: 1, modules: { dark: { defaultMode: 'light', brightness: 1.0, sites: {} } } }
```

**Rationale for a single root key:** keeps `chrome.storage.sync` item count low (quota: 512 items total, 8KB per item). A single ~8KB object can hold thousands of short eTLD+1 domains, and the 100KB total quota is comfortable.

## 5. Dark Mode Module

### 5.1 Mode Resolution

```
resolveMode(storage, domain):
  site = storage.modules.dark.sites[domain]
  if site === 'dark' | 'light': return site
  else: return storage.modules.dark.defaultMode
```

### 5.2 Content Script

Registered dynamically via `chrome.scripting.registerContentScripts` with `runAt: 'document_start'` for every domain where `resolveMode` returns `'dark'`. Injects a `<style id="omni-dark">` element:

```css
html {
  filter: invert(1) hue-rotate(180deg) brightness(var(--omni-brightness, 1));
  background: white;
}
img,
video,
picture,
iframe,
svg,
canvas,
[style*='background-image'] {
  filter: invert(1) hue-rotate(180deg);
}
```

Brightness is applied via a CSS custom property on `:root`, so the slider can update it live via `chrome.tabs.sendMessage` without re-registering the script.

### 5.3 Background Reconciliation

On any change to `omni.modules.dark` (from `chrome.storage.onChanged`):

1. Compute `nextEnrolled = computeEnrolledDomains(storage)` — the set of domains where effective mode is `'dark'`.
2. Diff against `previousEnrolled` (tracked in a worker-local variable, rehydrated on startup).
3. `chrome.scripting.unregisterContentScripts({ ids })` for removed.
4. `chrome.scripting.registerContentScripts([{ id, matches, js, runAt }])` for added.
5. For live tabs that match `toRemove`, tear down the filter on already-open tabs (see Section 12 — exact mechanism decided during planning).

**Asymmetry when `defaultMode === 'dark'`:** the enrolled set is `<all_urls>` minus per-site `'light'` overrides. Handled with one blanket registration + per-site exclusions (via `excludeMatches`).

### 5.4 Popup UI

Renders in the `Dark` tab of the popup shell:

- **Header:** moon icon + toggle switch (reflects current tab's effective mode).
- **Current site line:** `Current site: <eTLD+1>`.
- **Two buttons:**
  - `This site only` — sets `sites[domain]` to whichever value differs from `defaultMode` (or cycles: default → force-dark → force-light → default).
  - `All sites` — flips `defaultMode` between `dark` and `light`.
- **Brightness slider:** global, 50%–100%, live-updates active tabs.

## 6. Popup Shell

`App.svelte` renders a tab bar from `modules[]`:

```svelte
<script>
  import { modules } from '../core/registry';
  let active = $state(modules[0].id);
</script>

<nav>
  {#each modules as mod}
    <button onclick={() => active = mod.id} class:active={active === mod.id}>
      {mod.icon} {mod.label}
    </button>
  {/each}
</nav>

<section>
  {#each modules as mod}
    {#if active === mod.id}
      <mod.Popup />
    {/if}
  {/each}
</section>
```

With one registered module, one tab shows. No placeholders.

## 8. Build & Tooling

- **Bundler:** Vite + `@crxjs/vite-plugin`
- **Framework:** Svelte 5 (runes mode) + TypeScript strict
- **Domain parsing:** `tldts`
- **Output:** `dist/` — load unpacked in `chrome://extensions`
- **Dev:** `pnpm dev` → Vite HMR; reload extension in Chrome after manifest changes only
- **Prod:** `pnpm build` → static `dist/`

## 9. Testing

Vitest only (no E2E for v1, per user preference documented in agent memory). Coverage target: 80%+ on `src/core/` and `src/modules/*/service.ts`.

- `tests/setup.ts` — minimal `chrome.storage.sync`, `chrome.scripting`, `chrome.tabs` mocks
- `domain.test.ts` — eTLD+1 extraction (including tricky cases: `foo.github.io`, `co.uk`, IP hosts, `about:blank`)
- `service.test.ts` — `resolveMode`, `computeEnrolledDomains`, `diffRegistrations` as pure functions
- `storage.test.ts` — read/write round-trips, migration stub
- `registry.test.ts` — registry invariants (unique ids, required fields)

## 10. Known Limitations (v1)

These should be listed in `README.md` so users aren't surprised:

1. **CSS filter approach:** simple `invert + hue-rotate`. Looks ugly on sites with heavy gradients, shadows, or already-dark themes. Deferred to a future "smart mode" module upgrade.
2. **Fixed-position elements:** `filter` on `<html>` creates a new stacking context, which can break `position: fixed` sticky headers/footers on some sites. Known limitation of the approach.
3. **Subdomain granularity:** per-site rules use eTLD+1 only. `docs.github.com` and `api.github.com` share one setting. Fine for the common case; advanced users get subdomain overrides in a future release.
4. **No E2E tests:** v1 coverage is unit-only. Manual smoke test on install: enable on one site, reload, confirm dark; disable, confirm light.

## 11. Future Modules

The architecture anticipates these without implementing them:

- **Cookies** — per-site cookie editor / nuker
- **Smiley keyboard** — find and copy emoji through the keyboard
- **GDPR** — Auto dismiss cookie consent popups
- **Youtube customizer** — several youtube tweaks

Each will be specced separately. The module contract above is the stable API they plug into.

## 12. Open Questions Deferred to Plan Phase

- Icon assets — placeholder emoji for v1, real icons later
- Strategy for removing the dark filter from already-open tabs when a site is un-enrolled (reload tab vs message content script to tear down) — plan phase decision
